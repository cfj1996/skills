# 提测 Wiki 模板

## 创建位置

- 父级固定为 `提测文档` 下的月目录
- `提测文档` 页面 ID：`1150372234001008260`
- 月目录格式：`YYYY-MM`
- wiki 名称格式：`MM-DD: {简单描述}`
- 先确认目标月目录存在；不存在时，先在 `提测文档` 下创建对应月目录，再创建 Wiki
- 如果月目录下已经存在按模块分组的提测文档，必须插入到对应模块中，优先放到 `前端` 模块下
- 插入规则：
  - 如果已有 `前端` 模块且模块下已经有 1 个或多个 Wiki 条目，则按顺序追加到 `前端` 模块现有条目的末尾，保持编号连续
  - 如果已有 `前端` 模块但没有 Wiki 条目，则把当前 Wiki 作为 `前端` 模块的第 1 条
  - 如果没有 `前端` 模块，则创建 `前端` 模块并把当前 Wiki 作为第 1 条
- 当 `前端` 模块下插入时，最终结构应为：
  ```md
  # 前端
  1. xxx
  xxx

  2. xxx
  xxx

  3. {wiki模版}
  ```
- 如果 `前端` 模块不存在，则直接创建：
  ```md
  # 前端
  1. {wiki模版}
  ```

## 写入前校验

- 写入 TAPD 前，必须先把最终将要写入的 Wiki 正文完整展示给用户
- 正文必须完全符合下面的模板，禁止改成摘要版、风险摘要版、问题说明版或任意自定义格式
- 模板中的以下字段必须明确可填：
  - 序号
  - 项目名称
  - git 地址
  - 开发人员
  - 功能描述
  - 分支名
  - 影响范围
  - 测试人员
  - 环境
- 如果月目录、模块顺序、序号、测试人员、环境、分支名任一项无法确认，必须停止并向用户说明，不得自行猜测

## 模板

```md
{序号}. [项目名称](git地址) **!!#ff0000 更新服务!!**
- 负责人：{开发人员}
- 开发人员：
  - 前端：{开发人员}
- 内容：{功能描述}
- 代码分支名：{分支名}
- 影响范围：{影响的功能模块或页面}
- 测试人员：{测试人员}
- 环境：联团 老生产
```

## 约定
- `序号`: 第几个wiki
- `项目名称`：当前项目名；monorepo 场景取子包名称
- `git地址`：当前项目仓库地址
- `前端开发人员`：当前 TAPD 的登录人员 或 Bug 的 `de` 字段（开发人员）
- `功能描述`: 这个分支开发的功能描述
- `分支名`：统一使用真实 Git 分支名，不得使用 `short-id`
- `影响范围`：从 `task-plan.md` 风险评估或 `impl-summary.md` 汇总，多条使用序号
- `测试人员`：从 Story / Bug 的 `custom_field_two`（测试人员）获取；若为空则用 `reporter`（报告人）兜底
- `环境`：固定写 `联团 老生产`
- 提测完成后，Bug 评论正文必须固定为 `提测wiki：{wiki链接}`，不能写成说明段落、总结段落或多行说明

## TAPD Bug 字段说明

| 字段缩写 | 全称 | 含义 |
|----------|------|------|
| `te` | tester | 测试人员（提测时使用此字段） |
| `de` | developer | 开发人员 |
| `reporter` | reporter | 报告人（提交Bug的人） |
| `current_owner` | current_owner | 当前处理人 |
| `auditer` | auditer | 审核人员 |
| `confirmer` | confirmer | 验证人员 |

## 写入校验

- 先读取 `custom_field_two` 的真实值，再生成 Wiki 正文，不得用 `te`、`custom_field_2` 或自定义 fallback 猜字段。
- 先读取父级月目录正文，确认是否存在 `# 前端` 模块以及已有条目顺序，再决定序号和插入位置。
- 如果月目录正文为空，默认创建 `# 前端` 模块并把当前 Wiki 作为第 1 条。
- 如果正文中已经存在模块，必须采用增量追加，不得整页替换。
