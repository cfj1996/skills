# gitlab-map 使用约定

## 适用时机

- 阶段 4 / 5：提交前确认当前开发分支策略、基线或复用关系。
- 阶段 7：提交后确认当前变更满足合并到 `develop` 的条件。
- 阶段 8：合并前区分本轮提交范围和继承基线差异，避免把合法来源分支相对 `develop` 的历史提交误判为阻断。
- 阶段 8：提测 Wiki 生成前确认变更已经合并并同步到 `origin/develop`。

## 使用要求

- 仅接受 `gitlab-map` 返回的判定结果，禁止本地 `git` 命令替代。
- 分支校验结果需记录到当前计划载体的 `分支与合规检查` 小节；若没有计划文档，则必须在对话中明确记录：
  - 分支策略：`新建分支`、`复用线上 Bug 分支` 或 `复用需求分支`
  - 用户二次确认结果：`新建分支 + worktree`、`切换/复用已有分支` 或 `切换/复用已有分支 + 新 worktree`
  - 当前分支名
  - 校验时间
  - 基线分支
  - 来源分支是否合法：只能是 `origin/master` 或用户明确指定的功能分支
  - 当前 TAPD `short-id`
  - 复用分支时的原关联 TAPD / Story / Bug 线索
  - 是否通过
  - 未通过原因与修复建议
- 合并前校验必须记录：
  - 当前分支名
  - 目标分支 `develop`
  - 本轮提交范围
  - 继承基线差异：合法来源分支相对 `develop` 多出的历史提交，允许存在但不得进入本轮说明
  - 是否满足合并条件
- 提测前校验必须记录：
  - 当前分支最新提交
  - `origin/develop` 是否已包含该提交
  - 不通过时的阻断原因

## 不通过处理

- 新建分支来源不是 `origin/master` 或用户明确指定功能分支：停止提交和实现推进；若来源是 `origin/develop`，要求废弃该开发分支并回到合法来源重建。
- 复用分支关联关系不清：停止提交和实现推进，先确认当前分支是否仍承载该 TAPD/需求。
- 合并条件未通过：停止 GitLab 合并，先修正阻断项；但合法来源分支相对 `develop` 多出的历史提交不算阻断项，只记录为继承基线差异。
- 未合并到 `develop`：停止提测 Wiki 创建，要求先完成合并。
