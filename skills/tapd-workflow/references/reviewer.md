# 阶段四：代码评审

## 目标

核对当前实现是否满足需求，并决定是否可进入提交阶段。

## 独立 Agent 评审机制

- **禁止自我评审（既当裁判又当运动员）**：负责规划和编写代码的 Agent 不得自行出具评审结论。
- **强制交叉检查**：必须使用 `invoke_agent` 工具调用独立的子 Agent（例如 `generalist`，或按照环境内 `mr-code-reviewer` 的角色设定）来进行“背靠背”盲审。
- **调用方式**：在调用独立 Agent 时，必须将**本次变更的 Diff**、`plan.md` 的范围定义以及测试证据作为 prompt 发送。
- **工具调用失败策略**：如果 `invoke_agent` 或指定的子 Agent 无法使用，**严禁自行判定通过，严禁跳过独立评审流程**。此时必须暂停流程，并将真实错误报告给用户，等待平台工具恢复后重试。
- 只有独立评审 Agent 明确判断符合所有维度并没有低级错误后，才能认定为 `REVIEW_PASSED`。

## 评审维度

1. 需求一致性
2. 修改范围
3. 代码质量（重点检查低级错误、逻辑遗漏、未处理的边界条件）
4. 安全性

## 判定规则

- 独立 Agent 给出 `confidence >= 0.8` 通过
- 通过时输出 `REVIEW_PASSED`
- 不通过时给出明确的阻断项和修正建议；如需落文档，可复用现有评审或计划产物

## 评审范围

- 只看本次 diff
- 只看当前本轮声明、当前计划/执行记录和本次 diff
- 历史遗留问题仅作为建议项
- 评审通过后，只允许进入“提交后验证 / 合并到 develop / 准备提测 Wiki 草稿”阶段；不允许在评审阶段直接写 TAPD
