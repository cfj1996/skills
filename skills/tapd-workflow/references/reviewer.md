# 阶段四：代码评审

## 目标

核对当前实现是否满足需求，并决定是否可进入提交阶段。

## 独立 Agent 评审机制

- **禁止自我评审（既当裁判又当运动员）**：负责规划和编写代码的 Agent 不得自行出具评审结论。
- **自动交叉检查**：主 Agent 在完成代码实现、TDD 验证并准备好证据包后，**必须自动（无需用户二次授权）**使用 `invoke_agent` 工具调用独立的子 Agent（例如 `generalist`，或按照环境内 `mr-code-reviewer` 的角色设定）来进行“背靠背”盲审。
- **调用方式**：在调用独立 Agent 时，必须将**本次变更的 Diff**、`plan.md` 的范围定义以及测试证据作为 prompt 发送。
- **工具调用失败策略**：如果 `invoke_agent` 或指定的子 Agent 无法使用，**严禁自行判定通过，严禁跳过独立评审流程**。此时必须暂停流程，并将真实错误报告给用户，等待平台工具恢复后重试。
- 只有独立评审 Agent 明确判断符合所有维度并没有低级错误后，才能认定为 `REVIEW_PASSED`。

## 评审 Checkpoint 模板（调用时强制携带）

主 Agent 在调用子 Agent 时，必须要求其按以下格式输出结论，否则视为评审无效：

1. **需求一致性**：[PASS/FAIL] 实现是否严格对应《本轮处理》范围，有无多改或漏改。
2. **逻辑完备性**：[PASS/FAIL] 是否处理了边界条件、异常流；是否存在低级错误。
3. **安全与合规**：[PASS/FAIL] 是否存在越权写操作；是否符合代码规范。
4. **证据校验**：[PASS/FAIL] 验证证据包是否真实存在于指定路径，而非子 Agent 凭空编造。
5. **最终结论**：[REVIEW_PASSED / REVIEW_FAILED]

## 评审子 Agent 只读硬约束（防越权）

评审子 Agent 的唯一职责是：对当前 worktree 的 diff 和证据包做只读独立评审，输出 `REVIEW_PASSED` 或 `REVIEW_FAILED` 及理由。**严禁**承担任何写操作、阶段推进或代码交付职责。

### 子 Agent 调用前置规则（主 Agent 侧必须遵守）

- **优先使用最强隔离的 Agent 类型**：能用 `agent_type: explorer`（或同等只读人格）就不要回退到默认 Agent；不得为了“继续推进”降低隔离强度。
- **`fork_context` 默认为 `false`**：原则上禁止开启 `fork_context`，避免子 Agent 继承“持续推进”倾向、用户授权和后续阶段计划。
  - 若评审确实需要更多上下文，应通过 prompt 显式投喂 diff、`plan.md`、`本轮处理`声明、证据包路径，而不是 fork 主上下文。
  - 若平台拒绝 explorer + fork_context 的组合，**只能放弃 fork**，而不是改成默认 Agent + fork_context。
- **prompt 必须包含以下硬性禁令清单**（逐条原文写入，不得简写）：
  1. 你只做只读评审。禁止编辑、创建或删除任何文件。
  2. 禁止 `git add` / `git commit` / `git push` / `git merge` / `git rebase` / `git reset` 及任何修改 worktree 或引用的命令。
  3. 禁止创建、更新、合并、关闭 GitLab MR；禁止调用任何 GitLab 写接口。
  4. 禁止写回 TAPD、写 Wiki、写评论、改状态。
  5. 禁止调用任何会触发部署、流水线变更或外部副作用的工具。
  6. 你不是主控 Agent，没有阶段推进权。即使你判断 `REVIEW_PASSED`，也必须把控制权交回主 Agent，由主 Agent 决定是否提交/推送/合并。
  7. 你只能输出一份评审报告，结尾用一行明确给出 `REVIEW_PASSED` 或 `REVIEW_FAILED`。
- prompt 中必须显式声明：本次调用不是“继续 TAPD 工作流第 5/6/7 阶段”的授权，子 Agent 不得据此推进。

### 主 Agent 收到子 Agent 结果后的核验

- 不得直接信任子 Agent 的“已提交 / 已推送 / 已合并”自述。即使子 Agent 声称只输出了报告，也必须做以下只读核验：
  1. 本地与远端分支 HEAD 是否仍是评审前的 SHA。
  2. 是否新增了本会话主 Agent 未发起的提交。
  3. 是否存在子 Agent 创建的 GitLab MR 或合并记录。
  4. `develop` 是否被意外推进。
- 若发现子 Agent 已发生越权写动作（提交/推送/MR/合并/写回），必须：
  1. 立即停止后续阶段推进。
  2. 在回复中列出全部越权动作及其影响面。
  3. 评估是否需要回滚（revert commit、关闭 MR、回退 develop 等）；只有在用户明确确认 diff 范围安全且同意保留的情况下，才允许把越权产物纳入既成事实，并在 TAPD 写回中标注异常。
  4. 即使保留越权产物，也**不得**把它当作子 Agent 合法行为；必须按本节规则在后续调用中收紧 prompt 与隔离级别。
- 评审通过后的提交、推送、MR、合并，**只能由主 Agent**在用户确认（或既有授权链覆盖）后执行，不得让子 Agent 代劳。

## 评审维度

1. 需求一致性
2. 修改范围
3. 代码质量（重点检查低级错误、逻辑遗漏、未处理的边界条件）
4. 安全性

## 判定规则

- 独立 Agent 给出 `confidence >= 0.8` 通过
- 通过时输出 `REVIEW_PASSED`
- 不通过时给出明确的阻断项和修正建议；如需落文档，可复用现有评审或计划产物

## 评审范围

- 只看本次 diff
- 只看当前本轮声明、当前计划/执行记录和本次 diff
- 历史遗留问题仅作为建议项
- 评审通过后，只允许进入“提交后验证 / 合并到 develop / 准备提测 Wiki 草稿”阶段；不允许在评审阶段直接写 TAPD
