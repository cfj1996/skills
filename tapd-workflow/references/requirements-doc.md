# 需求文档提取规范

## 定义

- `原型文档`：通过 `chrome-devtools-mcp` 打开的蓝湖、MasterGo、Figma 等原型页面，是需求读取的源。
- `item-context.md`：从原型文档和 CLI 信息中提炼出的需求文档，是后续规划与实现阶段使用的工作产物，按类型保存在 `docs/bugs/item-{ID}/` 或 `docs/stories/item-{ID}/`。

## 触发条件

- 只要 TAPD 详情、评论或截图中出现原型链接，就必须进入原型文档读取流程。

## 提取步骤

1. 使用 `chrome-devtools-mcp` 打开原型链接。
2. 优先读取当前默认展示的需求文档、说明区域或页面注释。
3. 提取需求目标、关键交互、限制条件、验收线索和未明确项。
4. 将提炼结果写入 `item-context.md`，并标注来源为“原型文档默认展示的需求文档”。

## 多需求文档规则

- 原型中如果存在多个需求文档，只读取默认展示的文档。
- 不主动切换标签、下拉项或其他需求文档入口。
- 其他需求文档只记录“存在但未读取”，除非用户明确要求继续展开。

## 输出要求

- `item-context.md` 需要明确区分“原型文档原始信息”和“提炼后的需求结论”。
- 提炼内容至少覆盖：需求目标、关键交互、约束条件、未明确项。
- 如果原型可访问但默认展示的需求文档无法定位，需要在 `warnings` 中写明原因。

## 失败处理

- 原型链接失效、无权限或页面无法加载时，记录访问失败原因。
- 原型可访问但未通过 `chrome-devtools-mcp` 完成默认展示需求文档的提取，视为收集不完整。
